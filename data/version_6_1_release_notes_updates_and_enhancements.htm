1.4 ReportAnywhere

1.4.1 Custom Date Strategies

ReportAnywhere supports relative dates for report scheduling. This enables you to set report dates (such as the Beginning Date, End Date, Statement Date, etc.) relative to the run date of the report.

Note: Relative dates should not apply to the "run date", as "repeat options" handle that.

 

To enter a relative date on the report you are scheduling, right mouse click in the desired date field. When you do so, a date selection box will appear:

 

 

The custom strategy box enables you to set the report dates based on the date the report is run by the system according to the following rules:

d, m and y refer to the day, month and year of the current run date.

Each date component can be controlled separately.

Adjustments can be either relative (using + or -) or absolute (no sign).

d=last is a special case to compensate for the variable # of days in a month.

Omitting a date component is the equivalent saying 'date component=+0'  (i.e. today-->d=+0)

  A semi-colon must separate date components

Example:

Suppose you are scheduling the report to run every Sunday. To set the BEGIN DATE of the report to Monday, you could either select the "Tomorrow" option, or use custom strategy d=+1 (i.e. Monday). To set the end date to Wednesday, use the custom strategy d=+3 (i.e. Wednesday). The custom option enables a greater degree of flexibility

The following is a list of custom date strategy syntax and conventions.

 Option  Custom   

Yesterday d=-1

Today  d=+0

Tomorrow d=+1

PrevMMFirstDD m=-1;d=1   (first day of the previous month)

PrevMMLastDD m=-1;d=last   (last day of the previous month)

ThisMMFirstDD d=1    (first day of this month)

  d=15    (the fifteenth of this month)

ThisMMLastDD d=last    (the last day of this month)

NextMMFirstDD m=+1;d=1  (first day of next month)

NextMMLastDD  m=+1;d=last  (last day of next month)

PrevYYFirstDD y=-1;m=1;d=1  (first day of last year)

PrevYYLastDD y=-1;m=12;d=last (last day of last year)

ThisYYFirstDD m=1;d=1  (first day of this year)

  m=3;d=31  (end of first quarter, this year)

ThisYYLastDD m=12;d=31  (last day of this year)

NextYYFirstDD y=+1;m=1;d=1  (first day of next year)

NextYYLastDD y=+1;m=12;d=last (last day of next year)

3.4 Create EOM Unvouchered AP GL Entries

Purpose:

This function analyzes inventory history to identify items received, yet unvouchered and uses the P.O. costing to determine the expense that is expected when the invoices is ultimately received for the items.  When that is the case, it is sometimes desirable to account for items "received", that the client has NOT received a vendor’s invoice for. Typically, this would be an end-of-month entry to DR Expenses and CR an Unvouchered Payables Liability Account.  The account debited will the account listed as the Inventory Asset Account in the Inventory Group record. 

·      For purchases of ITEMS that are Inventory Items, one would expect that they would be attached to Inventory Groups that reference an Inventory account in this field - particularly if you were using the WO Inventory to GL feature in the Accounting Period.

·      The Inventory Group "MISC" is used for GEN Tab (non-item record purchases)...since there is no INVENTORY Tracking on such items, it would be appropriate to reference a COS Expense account in the "MISC" Groups inventory asset field, to effectively cause the item to be expensed by the process.

There are some users that do not wish to have a balance on their GL for inventory and wish to expense all purchases directly...this can be achieved by taking the approach recommended for the "MISC" group within "actual" groups" for items that are in the INVENTORY ITEM list.  If such an approach were taken it would be inappropriate and inconsistent to use the WO Inventory to GL feature in accounting periods.

Usage:

There are two different methods of accounting for unvouchered payables: by running this function periodically (preferred) or by flagging the Unvouchered Payables flag in the Accounting Period (deprecated in versions 6.1 and up). This function would be used ONLY when Unvouchered Payables is not being implemented for Accounting Periods!

Depending upon on your firm’s manner of doing business, you may only want to run this at year-end, but it can be performed at any month end.

Important Fields:

Post Unvouchered AP:

Period Id:
Enter the accounting period that you want to post to. The last day of the "posting" period is used to create DR to Expense accounts (using the Inventory Asset Account in the Inventory Group) & CR to UVP (using the GL Unvouchered AP Account in the Inventory Group).  If you enter an inactive accounting period, the system will warning you that it is inactive, but not prevent you from posting to it. This eliminates the need to temporarily re-activate a period. It also eliminates the possibility of a data entry person posting to the period while it is temporarily active.

Period Start: Period End: Period Active
These informational fields will be displayed when the Period Id is selected.

Reverse Unvouchered AP:

Period Id:
Enter the accounting period that you want to post the reversals to. The first day of the "reversing" period is used to make the net-opposite transaction, so that the Expense and Liability are only recognized on the "last" day {month end} financials.  At all other points in time, it would be expected that the UVP liability has NO balance.

Work Order Analytics - Changes in Version 9.1.611.052 and Above

The Work Order Escalation Statuses of Travel, Onsite and On Break have been added to support additional granularity in Scheduled Events and reporting.  These statuses were previously grouped together as Active. When a Work Order is escalated in this release, the Work Order Escalation Status will be set to Travel, Onsite or On Break (as applicable) if previously it would have been set to Active.  This will impact existing range Work Order Escalation filter and subfilter settings in DBAnalytics.  These filters should be reset manually when upgrading to SAMPro 9.1.611.052+ to ensure that your Work Orders are accurately displayed in DBAnalytics. Users may have to type the new escalation statuses into the filter field in lieu of selecting them from the look up, since they won’t be available to choose if not yet present in the data.

Setting the Escalation Status Filter In DBAnalytics

Setting In 9.1.611.051 And Earlier:  Active

Setting In 9.1.611.052 and Later: Active,Onsite,Travel,On Break

Note that an ‘Active’ Escalation Status is included in list of statuses so that Work Orders that have not been escalated with SAMPro 9.1.611.052 running will be displayed.  Work Orders with an ‘Active’ Escalation Status will eventually wash thru your system as they are escalated in the new build.

For example, the WO Performance Advanced Analytic may be used to show all workorders that are in progress, along with the unbilled costs they’ve accumulated.  You show and filter the escalation status instead of the technician status to see the current “overall status” of the workorder, and only one line per workorder.  This way the total at the bottom on the unbilled costs column isn’t multiplied times the number of technician dispatch lines.  Currently, your escalation status filter would include all open statuses including “Active.”  Upon receiving this update, all workorders that escalate to Travel, Onsite, or OnBreak would drop off the report unless you add these new escalation statuses to your filter.  This would be the case on any and all analytics you are currently using that are filtered by Escalation Status.

Taking Advantage of the New Escalation Statuses in DBAnalytics

The new Escalation statuses of Onsite, Travel and On Break allow you to set your filters to see only work orders with an Escalation Status of Travel (for example) or Onsite instead of all Active work orders as in previous releases. To do so, simply set the Escalation Status to Travel or Onsite as desired.

Escalation Statuses in SAMPro Escalation Profiles

The Statuses of Travel, Onsite and On Break have been added the Status list.  These statuses were previously grouped together as Active. Existing Escalation Profiles referencing the Active status will continue to be supported (for example, if the Work Order Escalation Status goes to Onsite, Escalation Profiles specifying the ‘Active’ status will still fire). New or existing profiles may specify Onsite, Travel, On Break or Active as statuses.

This enhancement allows users to apply a greater degree of granularity to upcoming scheduled events without loss of support for the ‘active’ status used in Scheduled Events.

If the Avalara Web Services are running, the following files will be open: SAMProNETFramework.dll and NewtonsoftJson.dll.  This means you will not be able to simply unzip the file with the image and drop them all in.

If the installed files SAMProNETFramework.dll and NewtonsoftJson.dll do NOT have a date DATE MODIFIED of 10/13/2020 the update must be done where users and services are logged off so that the .dll files are not open.  Remember, the VDEVW.exe.manifest, VDEVW.exe, v.exe.manifest, SAMProNETFramework.dll, NewtonsoftJson.dll and the Data-Basics.Avalara.Net.manifest (in the Data-Basics.Avalara.Net folder) must all work with the executable (v.exe) that you are installing or nothing will run.

GL Reporting Period

Define GL Reporting Periods for at least the next full year. If a ‘beginning’ period was used to load data, set the Start Date to the beginning of the loading year (not 01-01-1900) before upgrading.

The GL Account Type's Category field is used by DBAnalytics. Review all  GL Account Types and set the Category field appropriately if needed.

Work Order

Dispatch Board

The system allows multiple lines on the Work Order Technician tab with the same Tech Id and Date but a different Time Scheduled (TechAnywhere 3.0.048.006+).

AP Journal

 

Main Tab - GL Entity: Sets the GL Entity for manual checks, EFT and CC/DR Card transactions when posting to a Bank Account.  Previously it only assigned an Entity to the AP Control Account and Discounts Earned. Because of this change:

If InterCompany IS enabled, specify Corporate GL Entity for the company.

If InterCompany is NOT enabled, BLANK this field for manual checks, EFT and CC transactions to avoid Bank Reconciliation issues.  

Location Record

 

The system uses the GL Entity from the WO-INV line's Location record. Setting the GL Entity in the Location record ensures that the Credit GL Entity defaults when items are entered on the WO Inventory tab. This field is required if Intercompany Processing is enabled.

Pay Type

PTO Accrual Exclusion

(primarily for Province of Alberta, CA)

Fields are only applicable if Pay Type=PTO and PTO Accrual Method = Percent. Previously, only vacation wages were excluded from the vacation accrual wage base. The PTO Accrual Exclusions fields were added so the types of wages to exclude could be selected.

Important Upgrade Note: you must flag the applicable boxes in PTO records for these wages to be excluded from the accrual wage base. After upgrading to version 9.1 from 7.3 or before, carefully review your Pay Type records to see if any changes should be made.

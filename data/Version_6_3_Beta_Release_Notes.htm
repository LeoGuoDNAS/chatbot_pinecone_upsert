1.0 Important Data Conversion Notes on Job Templates

This version provides for the use of template Job, Clients and Site. Template jobs are prevented from having work orders (accidentally) attached to them, and provide a standardized starting point for the creation of new record structures.

Before data conversation, all of your current jobs, sites and clients that include the word template (not case sensitive – Template or TEMPLATE will also be picked up) in the name or id field will be flagged as a template record during conversion.

We strongly recommend that you review your data to verify that the records that should be flagged as templates include the word template in the name or id field. Client, Site and Job records that should not be flagged as templates should not include the word template in the name or id. To quickly ascertain which records this may impact, enter the characters %template% on the name and then the id fields of the client, site and job records. See sections 4.1, 4.6, 4.7, 4.8, 6.4 and 6.5 of the customer notes for details.

Setting Up Your Templates

Step 1: Decide on your Job Structures

Decide how your various template jobs should be structured in terms of cost codes, billing items and so on. You may wish to have several types of templates – for residential clients, commercial clients, and so on.

Step 2: Create the Client Template Record

Create a Client Template record in View Client Templates. The Template box must be checked. Only those fields that will be common to ALL clients generated from this Template should be filled in. All other fields in this template record (except those required to save the record) should be blank.

Step 3: Create the Client Site Template Record

Create a Client Site Template record in View Client Site Templates. The Template box must be checked. Only those fields that will be common to ALL sites generated from this Template should be filled in. All other fields in this template record (except those required to save the record) should be blank.

Step 4: Create the Job Template Record

Create a Job Template record in View Job Templates. The Template box must be checked. Set the Client and Client Site Id’s to the Client and Client Site Templates you just created.

Aside from the Client and Client Site fields, only those fields that will be common to ALL jobs generated from this Template should be filled in. All other fields in this template record (except those required to save the record) should be blank.

Step 5: Create the Job Billing Item Record(s)

Create at least one Job Billing Item record for each of your template jobs. The information on this record will be used for invoicing. All Billing Items associated with the Job will be cloned during the New Call Create and Clone Job Wizard processes. Typically, the Invoice and Billing Worksheet fields would be completed.

Step 6: Create the Job Cost Code Record(s)

Create at least one Job Cost Code Billing Item record for each of your template jobs. All Cost Codes associated with the Job will be cloned during the New Call Create and Clone Job Wizard processes.

Step 7: Create a Site Equipment Record

Create at least one Site Equipment record for each of your template sites. Be sure to complete the Job fields in this record with your Template Job.

1.0.2 Using your Template Files with Job Clone Wizard

You may select a Template Job in the regular Job Account Viewer. You will not be able to edit the Template, but you will be able to launch the Clone Job Wizard. When the Job has been cloned, it will appear in Job Viewer so that you can easily edit it and its associated records.

1.03 Using your Template Files With New Call Create

Once your template records have been created, using them in New Call is as simple as A-B-C.

Step A: Verify the Customer is Not Already in your Database

Use the New Call's Search Text feature to verify that the customer is not already included on your database: be sure to search by Name and at least one other parameter, such as phone number or street address.

Step B: Select the residential Client Site and Switch to New Call Create

Once you are sure that the customer is not in your database, select the residential site on the Client Site list in New Call.

With residential selected, click the Search button to turn Search Off and Create on.

Step C: Enter the New Client Site, Equipment and Work Order Information

In the New Call Create screen, type over the Client Site Id residential with your firm's id for this client site. Enter the work order and equipment fields as discussed in the New Call Create Help Topic. The Client Id and Name must remain blank: when you click the Add record button, the system will also create a new client record with the same Id as that of the new client site.

2.4 New Option for AP Journal Registry Entry (sys-viewer-vapjrnl)

WO Billing / PWP Arrangements

This registry option concerns you if you use Service Invoicing for WO billing and have PWP arrangements with your subs.  In previous builds, upon completion of the WO, an AR Invoice could not be generated with the sub's charges until the Vendor Invoice has been recorded in AP.  In this enhancement, the AR Invoice Id has been added to the AP Journal child tables.  During Vendor Invoice entry, the user may specify an AR Invoice Id to attach to the JCH entries, making it appear as though the charges have already been billed.  In cases where the sub PO was issued for a single WO and that WO has appeared on an AR Invoice, advise the user and preset the new field to the associated AR Invoice Id. Specifically:

1. When the APJ PO Id is entered, the PO is read to determine its WO Id.
2. If the PO has a WO Id, the JCH is queried to determine if the WO has been invoiced.
3. If the WO has been invoiced, the new logic is activated.

Conversely, if the APJ entry has no PO or the PO has no WO or the WO hasn't been invoiced, then the new logic is not activated. The following registry option is available to adjust the new behavior:

AttachCostsToARInvoiceForWO=always or never or prompt

The default behavior is 'always'.  This means that, whenever possible, the A/R Invoice Id should be set automatically. This behavior will occur if no registry entry is made.

Setting the option to 'prompt' means that the user will be informed that an A/R invoice for the WO exists and gives them the option of attaching it to the AP Journal costs.

Setting the option to 'never' means that the previous SAMPRO behavior can be restored.  In other words, the user will never know this change was implemented.

13.0 TechAnywhere RTD (Blackberry)

13.1 Overview

Introduction

This functionality enables your firm to send work order information from SAM Pro to your technicians in the field on RTD’s (Remote Technician Devices) such as Blackberries. This document provides background information on how information is passed between SAM Pro and the RTD, setup information, and technical details on managed list operations.

The registry entry needed to manage this functionality is described in Registry Entry RTD Remote Tech Device .

PM Tasks and Service Metrics processing added.

Mirifex Overview

Mirifex will be providing

- The client application running on the Tech Device.

- A transfer mechanism which will be used to transfer data between Sampro and the Tech device.

Shared folders will be used for the data transfer mechanism. Mirifex software will pick up outbound (from Sampro’s perspective) files from shared folders and transfer them to the Tech Device. Mirifex software will also pick up inbound files sent by the Tech Device and deposit them in shared folders.

All files transferred in both directions will be properly formatted XML files. The contents and format of these files will conform to Data-Basics specifications.

Work Order Transfer and Posting Cycle

Sampro (Publishing Of New Work Orders):
Whenever a Work Order is created or changed, SAM Pro’s Escalation Manger examines the Escalation Profile associated with the Work Order. Certain Escalation Events associated with the Profile can trigger the sending of the Work Order to the Tech Device.

When one of these events is detected, the Escalation Manager will queue up a Scheduled Event whose function is to initiate the writing of the outbound XML file representing the Work Order. That Scheduled Event will identify the Work Order Id and the Id of the Work Order’s Assigned Technician.

Scheduled Event Manager periodically examines its queue and identifies Pending events whose Scheduled Date and Scheduled Time has been exceeded. When it identifies the event queued as above, it invokes a routine which XML encodes the Work Order and writes the XML encoded Work Order to the Outbound XML folder. This folder is identified in the Sampro Registry.

The shared folder must be accessible by the Mirifex server. Mirifex software watches this folder for newly added documents. When Mirifex identifies a new work order (encoded as an XML document), it will automatically transfer the XML for the work order to the Tech Device corresponding to the Assigned Technician’s Email Address. It will then transfer the document from the Outbound folder to an Archive folder.

Tech Device (Receiving New Work Orders):
When the Tech Device receives a Work Order from Sampro, it places it in a folder which serves as an ‘open work order repository’.

Tech Device (Updating of Work Information by Technician)
The Mirifex Remote Tech application on the Tech Device is used by the technician to retrieve information about the work order, to make ‘Technician Entries’ such as labor and work completed against the work order, and to change the status of the work order.

Tech Device (Publishing of Work Orders Changes):
When the technician changes the status of a Work Order on the Tech Device, the Tech Device application will XML encode the Work Order and cause the XML to be forwarded to Sampro. A list of valid Status Codes and their associated status categories will be supplied to the Tech Device as a ‘Managed List’ (Managed Lists are explained below).

Sampro (Receiving Of Work Order Changes):
A Sampro scheduled event periodically executes the Sampro function ‘Srtddcmntpst’ (Start Remote Tech Device Document Post). Srtddcmntpst polls the appropriate folder for XML files incoming from Tech Devices. This folder is identified in the sys-rtd Sampro registry.

When Sampro finds a Work Order XML document in the Inbound Document Source Folder, it attempts to post that document to the corresponding Work Order. The Sampro Class ‘WOFromRTDPost’ implements the posting logic.

There are two types of posting mechanisms. The posting mechanism is determined by the status category corresponding to the WorkOrderStatus value founding the XML.

If the status category is ‘open’, then the document is treated as a ‘status change’ document. The new status information is extracted from the XML and the Work Order is updated accordingly. This involves searching the Work Order’s Tech tab for the line referencing the Assigned Technician; emulating the entry of the appropriate information on the line, and emulating the pressing of the Save key.

If the status category is other than ‘open’, then the document is treated as a ‘completion’ document. The Work Order’s status updated as above. In addition, the Work Order is updated with information from the XML’s TechEntries tag. Note that where this process requires adding entries to any of the Work Order’s child tabs, the new entries will be appended to any entries already in place.

Finally, the file containing the XML is either deleted or is moved to an archive folder as directed by the Sampro ‘sys-rtd’ registry ‘disposition’ entries.

Sampro (Managed List Updating)
The Mirifex application running on the RTD requires a number of ‘managed lists’. These are normally used to populate pull down lists displayed on the application’s forms. These lists are usually derived from Sampro tables. The contents of these lists may change from time to time. When a list changes, the new list must be uploaded to the tech devices. The shared folder used to hold the current set of managed lists is defined within the Sampro registry.

The Sampro class RTDManagedListWriter is used to generate XML for all the managed lists. It then compares the generated XML against that maintained in the OutboundManagedListFolder. If the generated entry is not already in the folder, or if it differs from the entry already in the folder, the newly generated XML document is written to the OutboundManagedListFolder.

The OutboundManagedListFolder must be visible to the Mirifex server software. The server software will recognize when a new document is created in the folder or an existing document is replaced, and cause the document is uploaded to all Tech Devices.

A Sampro scheduled event, periodically executes the Sampro function ‘srtdexprtmngdlsts’ which uses RTDManagerListWriter to update the OutboundManagedListFolder as necessary.

Sampro (Housekeeping)
If generated and received XML files are not deleted immediately after use, some mechanism must be provided to periodically delete files which are beyond a certain age. No such application currently exists and should be handled manually.

Sampro (Logging)
A logging mechanism can be enabled by appropriate entries in the ‘sys-rtd’ registry entry. A log entry is be generated whenever Sampro generates an XML file for publishing. A log entry is be generated whenever an XML file is retrieved from a Tech Device and posted. Any error conditions encountered in either of these operations will appear in the log.

Sampro (Error Reporting)
Any errors encountered by Sampro in the generating or posting of XML will be reported in the log.

Other Functions

Daily Work Order Export Function
This function may be used to export XML for all Work Orders whose Technician tabs meet certain criteria. These criteria are defined in the sys-rtd Sampro registry entry. In particular, the administrator may specify an effective date (‘today’ or ‘tomorrow’) and a list of Log Type Ids When this function is run, Technician Tab lines meeting the criteria trigger the generation of the corresponding outbound XML files. See Start RTD Daily Work Order Export for additional information.

This function would normally be automatically run on a daily basis via Scheduled Event Manager entries.

When pressed, this button will export an RTD XML file for the currently selected dispatch line. Note that this button is only enabled if the selected work order line is eligible for export (e.g., it is of the correct status, and is assigned to Technician with the correct web access).

13.2 Start RTD Daily Work Order Export

Purpose:

This function is designed to periodically export Work Orders to the Blackberry for Remote Tech Device users.
This function can be initiated periodically by the Scheduled Event Manager.

This function will create XML files in the FMMS Outbound Folder based on criteria stored in the sys-rtd registry entry.

Registry Entry:

The operation of this function will be controlled by parameters stored in the Registry Entry RTD Remote Tech Device .

Usage:

This function will operate as follows (assuming that the applicable registry entry has been set):

1. Scan the all lines of the Technicians tab of all work orders.

2. Disregard all lines for which any of the following are true:

The Scheduled Date does not match the DailyExportDate criteria. For the purpose of this test , ‘today’ and ‘tomorrow’ will be based the clock date in effect at the time the function is run.

The Log Type Id does not match one of the values specified by the DailyExportStatusCodes criteria.

The Technician is not Active.

The Technician does not have a Remote Device Type of ‘CE’.

3. For all technician lines not disregarded in the preceding step, the function will export an XML file for the corresponding work order and technician to the outbound folder for transfer to the Blackberry by Mirifex.

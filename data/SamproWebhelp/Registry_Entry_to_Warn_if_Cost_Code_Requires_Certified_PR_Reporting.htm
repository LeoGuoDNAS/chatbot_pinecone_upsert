Registry Work Order Options (sys-viewer-vwrkordr)

Purpose:

The options in this registry entry can be implemented to configure a variety work order related functionality.

Testing Your Registry Entries
The system does not verify the validity of registry entries. Be sure to test your registry entries after completing them to verify that they are functioning correctly.

Setting Values are Uppercase-Lowercase Sensitive:
The Setting Value is case-sensitive: be sure that you enter the value exactly as specified in the documentation. If the documentation specifies a value of true, do NOT enter True or TRUE.

Consolidate Imported Job History Records When Get Costs is Clicked in WO

With the following registry in place, costs previously imported from JCH into the Work Order Viewer are evaluated and “refreshed” or consolidated to decrease the amount of negative/reversing lines on the WO child tabs.

Id: sys-viewer-vwrkordr

Setting Key: ImportCostsRefreshAll    Setting Value: always or prompt or no   (default no)

The default setting is 'no'.  This preserves the existing behavior.

The preferred setting is 'Always'.  With this option in place, each time Get Costs is hit all WO lines previously imported from Job Cost History are removed from the viewer and evaluated along with any new costs coming in. Any old, and new lines that net to zero are eliminated. Old and new lines that are eligible for collapsing will be brought in as one newly revised WO line.

The 'Prompt' option gives the user the option of whether to consolidate with the following question:    

“Do you want to consolidate imported cost records?
Yes to consolidate. No to import without consolidating.”

*Note – separate transactions may not always consolidate depending on differences in data entry between the two transactions.

Example:

A Labor journal entry is imported into a work order, and saved after import. The same labor journal is then decreased by two hours, and Get Costs is hit on the work order.

With registry set to always, the original import is deleted from the work order and one line will be brought in with reduced amount.

With registry set to prompt, answer Yes for same result above, answer No and three lines are brought in - the original, the reversal and the new.

The example above is always eligible for collapse because it is from the same transaction. Separate transactions may not always qualify for collapse if for example, a description, date, cost category, etc. is different between the two. Keep this in mind when using this registry.

Credit PO Handling

One of the major differences between regular and credit POs is the posting of the Quantity Ordered and Quantity Received values.  On a credit PO, the Quantity Received is ignored and the Quantity Ordered is used in its place.  For example, on a regular PO the Quantity Received is posted to IMH as Quantity On-hand and to JCH as Actual Quantity.  For credit POs, the on-hand and actual quantities are derived from the Quantity Ordered.

Some customers may wish to use Quantity Received as an indicator that the vendor has actually credited them for the returned items rather than whether the Credit PO is vouchered.  By their definition, an open credit PO is one where the Quantity Received is less than the Quantity Ordered on one or more lines.  Due to the posting rules noted above, this distinction can only be made by evaluating the lines of the PO ... IMH and JCH can't be used.

The following registry entry changes the definition for open Credit PO’s to be those where the Quantity Received is less then Quantity Ordered:

Registry Id: sys-viewer-vwrkordr

Setting Key: CreditPOsMustBeReceived  Setting Value: false

The default value is true so as to preserve existing behavior.

Restore ‘Compute the Invoice’ option for Construction Type Jobs on Work Orders

Some clients use work orders for certain portions of construction-type jobs to manage their technicians; typically, these work orders are not billable. In previous releases, the act of 'finishing' these work orders by setting them to Invoice Ready presented the user with an option to 'Compute the Invoice?' when the work order was saved.  This is legacy code predating service compute-type jobs and billing. If the user answered Yes to the question, the system prepared the work order/job for billing by:

1. Creating a new Invoice Master record.
2. Preparing all relevant JCH records for billing.
3. Updating Billing Items.
4. Setting Job's Invoice Status to 'Ready to Print'.

None of the above is necessary for most systems, and results in an orphaned Invoice Master record. In this release, the default behavior now eliminates the 'Compute the Invoice? question, since the billing for these construction jobs will be done from Record Billing Adjustments.

Set WO Inventory Date for 'Picked' Required Resource Items

Data from the Required Resource tab is posted to the Inventory Tab when the Picked flag is set.  However, a problem occurs if the Date Required is not within the default Accounting Period range. A registry entry is available to adjust either the date or accounting period to keep them in sync.  When the defaulted Date and Accounting Period are NOT out-of-sync, NO adjustments are made to the defaulted values.  The rules noted below are used only when an adjustment is necessary.

Registry Id:       sys-viewer-vwrkordr

Setting Key: AdjustDefaultDateOrPeriod Setting Value: period or date

If period is selected, the Accounting Period table is queried for eligible Accounting Periods.  To qualify, the period must be Active and its date range must include the defaulted Effective Date.  When multiple periods are eligible, the one with the smallest Period Id is chosen to become the new Accounting Period for the current transaction.  If no eligible period is found, nothing is changed.

If date is selected, the Effective Date is adjusted to be either the Start Date or the End Date of the defaulted Accounting Period.  The Start Date is chosen when the current date is less than the Start Date.  The End Date is chosen when the current date is greater than the End Date.

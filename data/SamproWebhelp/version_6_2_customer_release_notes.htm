1.1 Document Scanning

SAM Pro now supports document scanning and archiving. The scanned document will be saved as a .jpg image. Multiple pages are supported. This new functionality may be used in a variety of purposes, such as scanning in vendor invoices as they are vouchered in AP.

1.1.1 Setting Up

1. Extract the compressed file samproscan.zip into the directory containing the SAM Pro executable file (v.exe). samproscan.zip contains the program acquire.exe that SAM Pro will call to acquire the scanned image.

2. All users that will be scanning documents must install a local scanner with TWAIN compatible driver software.

3. Create a directory on the local machine in which the scanned images will reside.

4. To enable scanning, you must have a registry entry for document archiving with an archive directory set. In addition, a directory for storing the scanned documents should be set.

Registry Id: 

 sys-document-archive

Contents/Additional Contents:

directory=<aDirectoryPathName>

This entry is required, and defines the directory where all document archives are stored.

  scandirectory=<ScanDirectoryPathName>

This is the local directory where the scan is stored.

defaultscanparms=parameters

This optional entry enables you to set the parameters used by the acquire.exe program. If the entry is not implemented, the default parameters are /Hide /Scan: the document will be scanned without bringing up the scanner driver box. Run Acquire.exe from the command prompt for a list of all parameters available. If you specify additional parameters you must also specify the default /scan and /hide options.

1.1.2 Scanning a Document

1. Access the SAM Pro record that you want to associate the scanned document with. For example, if you wish to scan the vendor invoice you have just received, the AP Journal entry you are using to voucher that invoice should be displayed.

2. Verify that the scanner is turned on, and that scanning is enabled in the record window. The record must be previously saved with no changes for scanning to be enabled!

3. Click the Scan Document button. The Scan Document Wizard window appears instructing you to ‘’Load Next Document to Scan. Insert the document you wish to scan in the scanner and then click the Scan Page button.

4. The Wizard will ask ‘Did Current Document Scan Ok’. After the scanner acquires the image and the Transferring Data box disappears, click the Yes button. If the Transferring Data box did not appear and the scanner did not seem to scan the document, click no and verify that the scanner is operational.

5. The Scan Document Wizard will prompt you to load the next page to scan. If you are scanning a multi-page document for this record, click Scan Page and repeat the scanning process. If you have scanned all pages for this record, click Finish.

6. The scanned page(s) are now attached to the record. They may be viewed by clicking the Attachment button located just above the Scan Document button. Note that multiple pages may be viewed from the ‘OtherFiles’ tab of the Document Viewer.

1.4 Email Groups

Purpose:

Email groups enable you to store list of email addresses, or lists of email groups to facilitate email distribution lists. An email group (surrounded by brackets) may be referenced, or stored in any email field in the system; the system will then email to all members of the group.

Usage:

When referencing an Email Group in an email field, you must surround the Email Group Id with brackets. For example, if you have created an Email Group with the Id sales, you would enter it in a SAM Pro email field as [sales] to indicate that it is a token replacement and not, itself, an email address.

The system evaluates the list of email addresses in the Email Group at the point of delivery, which is immediate when launched by clicking a "direct" email icon on a viewer...or...upon triggering of event when from scheduled events/reports.  The Email Group is evaluated and all email addresses contained in the group will be mailed.

Important Fields: Upper Section

Id:

Uniquely identifies this email distribution group. Do not include brackets in this field.

Name:
A brief description of this group (optional).

Default Domain Name:
If most or all of the users in this group share the same domain (e.g., @mycompanyname.com), you may enter it here and simply enter the individual user addresses in the lower section of the screen.

Important Fields: Addresses Tab

Email Address:
There are three different ways of entering an address on this line:

1. Enter the complete email address (e.g., sally@anycompanyname.com).

2. If the Default Domain Name is completed above, enter just the individual and the system will add the @ sign and the default domain for you (if you enter mary, and the default domain is set to @mycompanyname.com, the system will use generate the complete address mary@mycompanyname.com when the email is sent).

3. Email groups support nesting, so you may reference one or more existing Email Groups as token replacements within another.

Examples:

Email Group Id: sales  Default Domain: @mycompanyname.com
Addresses: jean, joe

 Note that both jean and joe are at mycompanyname.com. You may list each address on an individual line, or separate multiple addresses on a single line with a comma.

Email Group Id: dispatch  Default Domain: @mycompany.com
Addresses: charles,emma,tom,mary@anycompanyname.com

 Mary is at a different domain, so her complete address is specified.

Email Group Id: everyone Default Domain:
Addresses: [dispatch],[sales],prez@mycompany.com

 In addition to the president, everyone on the dispatch and sales groups will be included when the 'everyone' email group is referenced.

Since some of these records already contained contact information, the conversion will create "contacts" for existing structures that had data elements that are now maintained in the contact record (not all of them originally had these data elements, so not every structure will lead to new contact records). Records created during conversion will be marked as such. We suggest that you review your contact data after the conversion is run. It is possible that more than one Contact record may have been created for a single individual. In this case, all records should be edited to point to the desired contact record. Once the References tab an extraneous Contact record is blank (e.g., no other records reference it), the extraneous Contact record may be deleted.

Escalation Manager and related functionalities now goes to the referenced contact record to perform the email address lookups.

Since the Standard Cost is used if the system cannot locate a more recent / precise cost, it is very important for this field to be completed in the Inventory Item record.

2. Standard from Inventory Item [unless the RevenueFromStandardCost registry option is enabled]

9.13.1 Enhanced Support for Work Order Technician Token Replacement

Prior to this change, Escalation Profiles were limited to a replacement token capable of extracting the Work Order Id (^wrkordr_id^) and calling it the PO#.  This was possible because POs created via New Call were given Ids that matched the Work Order Id.

A new token-type has been implemented for the purpose of providing access to WO Tech line data.  Tokens beginning with 'wrkordrtchncn:' (NOTE the colon) employ special SQL logic.  The general format to use is...

  ^wrkordrtchncn:{database column name}^

The token for showing the WO Tech line PO Id in an email message is ^wrkordrtchncn:prchseordr_id^.  Other tokens can be used to extract data from the Technician and Purchase Order records.

Note that token replacements happen when the scheduled events get released for execution.  That means the ^date^ and ^time^ tokens are more like date and time executed.  When necessary, the Work Order’s date and time created can be extracted with the WO's ^wrkordr_dte_opnd^ and ^wrkordr_tme_opnd^ tokens.

1. Determine the Purge Thru date by applying the DaysToSave setting to the current system date.
2. Purge ALL cancelled and executed events regardless of work order status. 
3. If IncludeClosedWorkOrders is True, eliminate all remaining events for closed work orders.  An example would be a contingent Alarm that never went 'active' and has an Event Status of OnHold.
4. Clear the Escalation Profile Rn for qualifying 'closed' work orders.  This is done to prevent the creation of additional escalation events in the event someone edits a 'closed' work order

12.2 Web Profile Viewer

Purpose:

This record enables you to construct a ‘profile’ that will enable your web users to only ‘see’ those sites that they should be able to access. This record also enables you to implement custom web graphics and CRMAnywhere formatting.

Usage:

Referenced in records: Client Site, Web User.

The Web Profile Viewer supersedes the older Web Security record. Although version 6.2 supports both Web Profiles and Web Security, you are strongly urged to migrate to the Web Profile/Web User standard.

Example:

Typically, a Web Profile record would be made for a logical group of Web Users. For example, if your firm serves BigBank customers, you would create a Web Profile, BigBank, for CRMAnywhere users from BigBank. If BigBank had branches in several regions, you might create a Web Profile for each region, or create one BigBank profile, and define the regions/area via the custom values. You would then create a Web User record for each BigBank customer, specifying which BigBank Sites this user could access. For example, a BigBank branch manager would be restricted (in the Web User record) to only his store, while a BigBank area manager would be able to view information on all branches in her district. Click Web Users Viewer for further information.

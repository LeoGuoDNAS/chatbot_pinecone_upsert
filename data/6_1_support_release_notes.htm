Previously Undocumented Feature

~ (Global Not)

The tilde (~) is a NOT operator. When the first character is a ~ the rest of the filter indicates which to exclude. For example, ~a,p,m will include all values except a, p or m. This operator allows for more flexibility than ! or <>.

NoRnMap: If the NoRnMap is present {in the Options section}, RnMapping is disabled. RnMapping is a mechanism for improving performance by generating a map of [key column] values to record numbers in the database table. When you viewing a record or journal entry, the Tools menu now includes the ‘Change Log’ option. 1.2 Logon History Table

A logon history table, lgnhstry, is included in SAM Pro databases, and may be accessed via SQL. This table records the User Id, Logon Date and Time, Engine, Logoff Date and Time, Success (of logon), and Windows Logon name (Windows Logon name is not supported in Windows 98 and ME).

1.3 Splash Bitmaps

When a system is upgraded, the existing splash bitmap (v.bmp) will be copied to a version with the date included in the filename (_yyyymmdd.bmp). If present, startme.exe will reference this bitmap instead of v.bmp. To have a new splash screen appear, delete the v – dated bitmaps and replace the existing v.bmp bitmap with the new splash screen. When startme.exe is run, it will create the appropriate copy of the bitmap and display that.

1.4 Logon Time Monitoring

The new startme.exe files distributed with versions 6.1.358.025 and 6.2.378.beta_0610 (and later) support a new logging feature that enables our support representatives to monitor the time it takes to execute the various phases of the login process.  Monitoring could be implemented if, for example, logon’s seemed to be taking an unreasonably long time for one or more users at a client’s site.

To implement this option (with an appropriate version of the startme.exe file), you must include a new command line option. To do so, change the startme shortcut to read:

startme.exe -timeLogon

The results can be found in the SAMPRO folder in a file named "<SAMPROUserId>~Logon.txt" that will be automatically generated by the system.

2.4 Registry Entry to Control Type Ahead in Id Fields sys-id-autofill

Purpose:

This registry entry enables the default type-ahead (also known as Autosearch) behavior to be altered on a table-by-table basis.

Background

In previous releases, Type Ahead was enabled in most Id fields: you would start entering the Id, and the system would complete it for you as soon as it located a ‘match’. However, due to performance issues, this functionality was disabled for the job, client, and client site within the image itself. This hard-coding of type-ahead meant that it could not easily be tuned to specific user implementations.

Changes in this Release

In the absence of this registry entry, versions 6.1.358 and up observe the following default behavior:

No Type Ahead By Default:

Numeric Id’s generated by the system, including Journal Id’s, PO’s, Work Orders, Client Orders, Quotes. In these cases, type ahead does not gain the user anything because the id is most likely not defined until the last digit is typed in.

The special cases of Job Cost Code, Job Billing Item, Client Site Equipment (because these records are children of the Job and Client Site records (respectively)).

Type Ahead Turned On By Default

All other records with alphanumeric Id’s.

Important Fields:

Id:

 sys-id-autofill

Contents / Additional Contents:

 tableid=true (turns on type ahead) or

tableid=false (turns off type ahead)

where tableid is replaced by the function id of the table.

Example:

Assume that your sys-id-autofill registry entry has the following contents:

clntste=false

wrkordr=true

In this case, all the type-ahead defaults would be observed except that type ahead would be turned off for client site id fields, and on for work order id fields.

5.1 Integration of Web Front-End Sign On with Extranets

Purpose:

This enhancement implements a web front-end mechanism to enable the users of extranets (who are logged into the extranet) to transition smoothly to parts of Sampro's CRM/CSR web interface without having to log in again.

This feature extends the Login web RPC to allow extranet users to pass on a pre-generated Session ID string (and a timeout length), along with the username and password (for authentication by SAM Pro). The idea is that when external users log into their extranet and receive a session id, they can also log that session id with SAM Pro, so it can be used at any time during their browsing session.

Usage:

An external application (within the customer’s extranet) must be able to generate a valid session number and call to SAM Pro’s web front-end using a regular web request (http GET) to the address:

http://domain:port/action?rpc=registerSession&session=[session-id]&user=[username]&password=[password]&timeout=[number_of_seconds]

Parameters:

[session-id]  -- (Required) an alphanumeric session id, generated by the external application.
[username]    -- (Required) a valid Sam Pro web user (either a technician with 'client' or 'csr' web access, or the new style Web User (version 6.2 only)
[password] -- (Required) a valid password for the user, plain text (either from the technician record with 'client' or 'csr' web access, or the new style Web User (version 6.2 only)).
[number of seconds]  -- (Optional, defaults to 10000 if not present) session timeout period, in seconds.

Note that the external application must be able to generate the session id. The extranet log in and the SAM Pro web logins should be identical.

Login In Results:

If the username, password or session id is missing, an HTTP

400 Bad Request
status code is returned, along with an explanation for the error, in the http body.

If the username or password is invalid, disabled or expired, an HTTP

403 Forbidden
status code is returned, along with an explanation in the http body.

If neither of the previous results is returned, the 'registerSession' rpc method then registers the session with the web server (and creates the necessary temporary tables, for technician users).

If the operation is successful, an HTTP

200 OK
status code is returned.

Accessing the SAM Pro Web Interface After Log In:

The Sam Pro web interface can now be accessed at any time (within the timeout period) by the external application by, linking to any valid rpc action.

Example: To link to the Main menu:

http://domain:port/action?rpc=main&session=[session-id]

(the session id has to be one that was previously registered).

Logging Off / Ending the SAM Pro Session:

The external program can either let the session time out (at which point it's removed), or
can call the logout rpc when the user exits the extranet:

http://domain:port/action?rpc=logout&session=[session-id]

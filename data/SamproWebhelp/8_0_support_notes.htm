4476  Inventory Item

New Field: Obsolete

If the existing Inactive flag is checked, this item may not be entered on Purchase Orders, but may be entered on Work Orders and Client Orders (to use up existing stock).

If the new Obsolete flag is also checked, you may not enter this record on anything – Work Orders, Purchase Orders, Client Orders, Job Journal, etc.

This field effectively produces a 2-tier system for managing inventory. Inactive Items would be ones that you didn't want to purchase, but were still in stock. Once the item was depleted, it could be flagged as Obsolete so it could not be entered on anything.

The system will automatically set the Inactive flag when the user sets the Obsolete flag, but not the other way around. Clearing the Obsolete flag does not clear the Inactive flag.

An sql that support can run to set the Obsolete flag if Inactive is set at the customer's request is attached to this case.

4483  Log Form Cycle Codes

Important Note on Metrics/Log Form Lines: when a work order referencing a log form is created in SAM Pro, metrics (log form lines) will only be written to the Work Order Metrics tab if the Technician is not a TechAnywhere tech (e.g., Remote Access is set to None) or if no Tech is specified. If it is initially assigned to a technician with remote access, metrics will be uploaded to the technician’s device, and then posted to the work order in SAM Pro when they have been completed in the field.

Log Form Lines Viewer

CC: Cycle Code
This field may be used to control which metrics are written to the Work Order Metrics tab when Service or PM Work Orders are generated. Note that Log Form field in the Client Site Equipment record determents which list of metrics is used.

When a Service Work Order is created in SAM Pro’s New Call, the system will check the Cycle Code Filter in the Work Code; if this field is completed, only those log form line metrics whose CC: field corresponds to the filter or is blank will be written to the Work Order (for Techs without remote access) or uploaded to the Technician’s device.

When a PM Work Order is created, the system will check the Cycle Code Filter in the Client Site Equipment PM Schedule. Tip: If a log form metric should ALWAYS be recorded, leave the CC field blank.

This filter is used by Sam Pro and TechAnywhere to filter Work Order Log Form Metrics lists. Note that Log Form Id field in the Client Site Equipment record determines which list of metrics is used. The Work Code in the Work Order header is used for metrics filtering purposes for Service Work Orders.

For PM Work Orders, the Cycle Code Filter (if completed) in the PM Schedule Tab of the Client Site Equipment, or the Schedule Tab of the Master Contract will override the Work Code’s Cycle Code Filter.

4362  Tech Anywhere

Previously, data from TechAnywhere can be posted to the default accounting period, even if the effective date is outside of the period range. A problem occurred when the user attempted to bill the WO and encountered an error indicating that the effective date was out of period range. A registry entry is now available to adjust either the date or accounting period to keep them in sync. 

Id: sys-techanywhere

When the defaulted Date and Accounting Period are NOT out-of-sync, NO adjustments are made to the defaulted values.  The rules noted below are used only when an adjustment is necessary.

Setting Key: AdjustDefaultDateOrPeriod

Setting Value: period

The Accounting Period table is queried for eligible Accounting Periods.  To qualify, the period must be Active and its date range must include the defaulted Effective Date.  When multiple periods are eligible, the one with the smallest Period Id is chosen to become the new Accounting Period for the current transaction.  If no eligible period is found, nothing is changed.

Setting Value: date

The Effective Date is adjusted to be either the Start Date or the End Date of the defaulted Accounting Period.  The Start Date is chosen when the current date is less than the Start Date.  The End Date is chosen when the current date is greater than the End Date.

NOTE: the current TechAnywhere post behavior does not change unless the new registry setting is turned on. .

When the first user (of a particular OS) accesses a new image, the underscore version of the v.exe runtime will be generated with the user’s OS designation. For example, if you have XP and Windows 7 users, you would expect to see two runtime images:

4003  AP Payment Approval (Kaback mods)

When running under the KABACK cus; there are changes to the screen displays for Start AP Payment Approved List PM (approver1), Start AP Payment Approved List (approver2), View AP Date To Pay List (approver3).

They all have been reconstructed to look very similar with slight variations. New fields add. Functionally, a change was made that now we track how much each approver releases for payment. This is done so that we now know what approver's list to send the transaction back to after payment processing.

An example may better describe this functionality.
Vendor's invoice is for $1000.00. Approver 1 approves $800, Approver 2 is at $800, Approver 3 pays $500. In this example Approver 1 and 2 will not see this invoice on their list again until Approver 3 pays the remaining $300. Example 2: Vendor's invoice is for $1000. Approver 1 approves $800, Approver 2 is at $800, Approver 3 pays $800. Now this transaction will be back on Approver 1's list to allow him to approve the remaining $200.

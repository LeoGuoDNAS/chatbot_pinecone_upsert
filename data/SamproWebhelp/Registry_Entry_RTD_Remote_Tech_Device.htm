Registry RTD TechAnywhere Remote Tech Device (sys-rtd)

Purpose:

This registry entry controls the operation of the Start RTD Daily Work Order Export function used to send work orders to Blackberry-type devices. It also controls how out- and in-bound files are handled.

See RTD TechAnywhere Blackberry Setup for additional information on setting up this functionality.

Important Fields:

Id: sys-rtd

Outbound File Settings

Version 7.1 Additional Contents: OutboundWOFolder=path

Version 7.2 Setting Key: OutboundWOFolder Setting Value: path

Identifies the pathname of the folder in which generated XML documents representing Work Orders will be placed.

Version 7.1: OutboundWOFileName=NewWO<wrkordr_id>-<wrkordr_stts>-<tchncn_id>.XML

Version 7.2 Setting Key: OutboundWOFileName
Setting Value: NewWO<wrkordr_id>-<wrkordr_stts>-<tchncn_id>.XML

This option serves as the model for the filename of XML files generated for outbound Work Orders. In the model

The string ‘<wrkordr_id>’ will be replaced by Id of the generated Work Order

The string ‘<wrkordr_stts>’ will be replaced by the (internal) escalation status of the Work Order

The string ‘<tchncn_id>’ will be replaced by the ID of the corresponding Technician

All other characters will be directly copied.

Version 7.1 Additional Contents: OutboundManagedListFolder=path

Version 7.2 Setting Key: OutboundManagedListFolder Setting Value: path

Identifies the pathname of the folder in which generated XML documents representing Managed Lists will be placed.

Inbound Document Settings

Version 7.1 Additional Contents: InboundDocumentSourceFolder=path

Version 7.2 Setting Key: InboundDocumentSourceFolder Setting Value: path

Identifies the pathname of the folder in which inbound XML documents are to be found.

Version 7.1 Additional Contents: InboundDocumentDisposition=value

Version 7.2 Setting Key: InboundDocumentDisposition Setting Value: value

Indicates what is to be done with inbound documents after they have been processed by Sampro. Possible values are ’delete’ and ‘archive’. If ‘delete’ is indicated, the document will be discarded after processing. If ‘archive’ is indicated, the document will be moved to an archive folder after processing.

If no InboundDocumentDisposition is provided, ‘archive’ will be presumed if an InboundDocumentArchiveFolder key is present and ‘delete’ otherwise.

Version 7.1 Additional Contents: InboundDocumentArchiveFolder=path

Version 7.2 Setting Key: InboundDocumentArchiveFolder Setting Value: path

Identifies the pathname of the folder in which incoming XML documents will be moved after they are successfully processed by Sampro.

Version 7.1 Additional Contents: InboundDocumentRejectFolder=path

Version 7.2 Setting Key: InboundDocumentRejectFolder Setting Value: path

Identifies the pathname of the folder in which incoming XML documents will be moved after their processing by Sampro generated an error or a warning.

Log File Defaults

Version 7.1 Additional Contents: LogMode=value

Version 7.2 Setting Key: LogMode Setting Value: value

Indicates how messages sent to the log are to be processed.

A value of ‘off’ indicates that no logging is to be performed.
A value of ‘on’ indicates the log messages are to be written to the log file defined by the ‘LogPath’ key.
A value of ‘debug’ indicates that the log messages are to be displayed via a MessageBox.

If no LogMode is provided, ‘on’ will be presumed if a ‘LogPath’ key is present and ‘off’ otherwise.

Version 7.1 Additional Contents: LogPath=path

Version 7.2 Setting Key: LogPath Setting Value: path

Identifies the pathname of the file in which log messages are to be written.

Status Code Mapping

Version 7.1 Additional Contents: StatusCode=status , category , externalname

Version 7.2 Setting Key: StatusCode Setting Value: status , category , externalname

Identifies the Log Type Ids which you expect to see in inbound and outbound XML files.

There may be as many of these entries as required. At least two values, separated by commas, are required for each entry. Internally, the contents of the StatusCode entries are used to populate the WorkOrderStatusCode Managed List file.

The first is a valid SAM Pro Log Type Id (waiting service, unacknowledged, cancelled, etc.)

The second is the category of the status (‘Open’ or ‘Closed). A ‘Closed’ category informs SAM Pro to post status changes and technician entries (e.g., of hours worked or materials used) from the inbound XML to the Work Order in SAM Pro. For any other category (Open, at this point) only the Log Type Id for the appropriate technician is updated (on the Work Order’s Technician Tab).

An optional third value identifies an optional ‘external’ name for use in the Remote Device. If the third value is omitted, it defaults to the first value.

Daily Export

Version 7.1 Additional Contents: DailyExportDate = ddddd

Version 7.2 Setting Key: DailyExportDate Setting Value: ddddd

Any other value (or absence of this parameter) will result in no daily export being performed. This registry entry is used within the optional Daily Work Order Export function to identify which Technician tab lines are used to generate the outbound XML files. Valid values are ‘Today’ and ‘Tomorrow’.Only lines having Scheduled Dates matching this parameter trigger XML generation.

Version 7.1 Additional Contents: DailyExportStatusCodes Setting Value: statusCodeList

Version 7.2 Setting Key: DailyExportStatusCodes Setting Value: statusCodeList

where statusCodeList is a list of Sampro status codes (Log Type Ids, actually) separated by commas. Only those Work Orders with Technician lines set to these statuses (and with dates corresponding to today or tomorrow, as specified above) will be exported. This registry option is used within the optional Daily Work Order Export function to identify which Technician tab lines are used to generate the outbound XML files. This entry consists of a list of Log Type Ids separated by commas. Only lines having a Log Type Id matching one of the values in the list will trigger XML generation.

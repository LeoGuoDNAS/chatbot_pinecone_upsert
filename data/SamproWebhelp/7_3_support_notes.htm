4362  Tech Anywhere

Previously, data from TechAnywhere can be posted to the default accounting period, even if the effective date is outside of the period range. A problem occurred when the user attempted to bill the WO and encountered an error indicating that the effective date was out of period range. A registry entry is now available to adjust either the date or accounting period to keep them in sync. 

Id: sys-techanywhere

When the defaulted Date and Accounting Period are NOT out-of-sync, NO adjustments are made to the defaulted values.  The rules noted below are used only when an adjustment is necessary.

Setting Key: AdjustDefaultDateOrPeriod

Setting Value: period

The Accounting Period table is queried for eligible Accounting Periods.  To qualify, the period must be Active and its date range must include the defaulted Effective Date.  When multiple periods are eligible, the one with the smallest Period Id is chosen to become the new Accounting Period for the current transaction.  If no eligible period is found, nothing is changed.

Setting Value: date

The Effective Date is adjusted to be either the Start Date or the End Date of the defaulted Accounting Period.  The Start Date is chosen when the current date is less than the Start Date.  The End Date is chosen when the current date is greater than the End Date.

NOTE: the current TechAnywhere post behavior does not change unless the new registry setting is turned on. .

4236  Service Channel – Change to File Naming Logic

When Sampro receives a new service request XML from ServiceChannel, it creates a file containing the new service request in its Inbound folder. Logic exists to prevent naming conflicts within the Inbound folder.Under rare timing situations, this file naming scheme can result in an inbound file getting the same name as a previously received service request which has already been moved to the Archive folder. This creates an error when an attempt is made later to move the newer file to the Archive folder after it is posted.

Previously, the naming logic took into account the possibility that another service request had previously been received at the same date and time (to seconds). It did this by checking the Inbound folder for file naming conflicts.This logic has been expanded so that the Archive and Reject folders are included in the naming conflict resolution.

Note that this problem only occurred when new service requests were being received within the exact same second that the posting routine was moving files from the Inbound to the Archive folder. Since the posting routine only runs every 5 minutes or so, this was a rare problem.

4042  Service Channel - New Registry Entries for Multi-Company Use

Purpose:

Invoices are uploaded to ServiceChannel by sending them an email with an attached spreadsheet. For each work order, the spreadsheet includes a userId and a password. Previously, the userId and password are derived from the Sampro Registry sys-fmms-vendor-servicechannel, supporting only a single userId and password contained in the registry entry. This results in the sending of the same userId/password pair for every work order regardless of the Sampro 'company'.

This procedure has been enhanced so that userId/password used is dependent on the corresponding work order. This release provides a mechanism for implementing multiple userId/Password pairs when uploading invoices to ServiceChannel. The technique used is to provide for PIN based overrides in the Sampro ‘sys-fmms-vendor-servicechannel’ registry entry.

It work likes this:
•    The entire registry entry is retrieved from the Sampro Registry for the ‘sys-fmms-vendor-servicechannel’ key as usual.
•    The entire registry entry is retrieved from the Sampro Registry for the ‘sys-fmms-vendor-servicechannel-pin-XXXX’ key (where ‘XXXX’ is the pin identified by the work order).
•    The two entries are merged. Where there are duplicate entries, the second pin based registry takes precedence.

To implement the correct userId/password uploading for multi-company users, you must create a ‘sys-fmms-vendor-servicechannel-pin-XXXX’ for each company’s pin. The only lines needed are the InvoiceUID and InvoicePWD lines. For single company users, the invoiceUID and InvoicePWD lines can remain in the normal 'sys-fmms-vendor-servicechannel' registry entry.

Important Fields:

Id: sys-fmms-vendor-servicechannel-pin-XXXX

Where XXXX is replaced by the PIN provided by Service Channel. Note that individual registry entries must be created for all applicable PIN’s.

Required Settings for Multi-Company Use:

Setting Key: InvoiceUID  Setting Value: User Id provided by Service Channel

Setting Key: InvoicePWD  Setting Value: User Password provided by Service Channel

Make sure to specify the User Id and Password appropriate to the PIN specified in this registry entry.

3763  Viewers (general)

This case adds reporting of SQL 'Duplicate Key' errors generated during child table processing. It has to do with the user attempting to enter a duplicate key when a database constraint exists. One simple scenario that demonstrates the problem is this ... schedule the same tech multiple times on the same date and work order.

If the database is structured properly, there will be a UNIQUE CONSTRAINT on the wrkordrtchncn tables wrkordr_rn, tchncn_rn and wrkordrtchncn_dte_schdld columns.  This constraint causes the second of the duplicated lines to fail on both the insert and the update.  The failure was suppressed and never explicitly reported to the user.

The end result in the above scenario was the 'disappearance' of the second (third, fourth, etc) of the duplicated lines.  When the save completed and the viewer refreshed, the duplicated lines were gone.

In this release, the error is reported to the user. For example:

CHILD ROW UPDATE FAILED …

Error

-196

[SQL Server]Cannot insert duplicate key row in object ‘dbo.wrkordrtchncn’ with unique index ‘WorkOrderTechniciansByTechnicianByDate’.

3769  Database Conversion

This release implement incremental Commits when creating Foreign Keys and Indexes during SQL Server DB Conversion. Note that errors in creating Foreign Keys are the result of data integrity issues that already existed before the conversion executed. The issue that was addressed is that the failure to create one or more FKs in a given table resulted in no FKs being created.  This is because the commit wasn’t executed until all FKs for the table had been made.  If even one fails, a rollback was executed instead of a commit. In the release, the commits are made incrementally so that all will not fail due to one.

3697 Logon History

These code changes were made in response to an issue that Sue encountered during an on-site visit.  Derek helped her discover the cause and they were able to work around it by changing the ODBC definition.

The issue resulted from the ODBC definition connecting to a database that had a name in excess of the ten character limit imposed by the column named lgnhstry.lgnhstry_dtbase_id.  The problem only arose when switching Security Identities.

I discovered that the initial login process posts the NAME OF THE DATA SOURCE to the lgnhstry_dtbase_id column.  When the security switch happens, the NAME OF THE DATABASE pointed to by the data source is used.  The database name is actually more informative so I changed the initial login process to use it.  This also resulted in the need for a larger column so lgnhstry_dtbase_id was expanded from ten to twenty characters.

A general fix was implemented in the Smalltalk code in an attempt to handle the truncation issue before it happens, thereby preventing fatal errors.  For example, if an ODBC definition points to a database with a name that exceeds the new 20-character limit, the value gets truncated to the 20-character limit before submitting it to the database.  This results in the first twenty characters being posted to lgnhstry_dtbase_id and prevents the 8152-type error.

Also note:

1. Sybase always did the truncation instead of reporting the fatal error.  This change should make SQL Server act like Sybase.

2. An attempt was made to properly set the lgnhstry.lgnhstry_w32_lgn_usr_nme which appears to have been broken by changes to the Windows Registry in Windows 7.  The revised code worked as expected on my machine and Jean's.

3427 Work Order viewer and Credit PO’s

One of the major differences between regular and credit POs is the posting of the Quantity Ordered and Quantity Received values.  On a credit PO, the Quantity Received is ignored and the Quantity Ordered is used in its place.  For example, on a regular PO the Quantity Received is posted to IMH as Quantity On-hand and to JCH as Actual Quantity.  For credit POs, the on-hand and actual quantities are derived from the Quantity Ordered.

Some customers may wish to use Quantity Received as an indicator that the vendor has actually credited them for the returned items rather than whether the Credit PO is vouchered.  By their definition, an open credit PO is one where the Quantity Received is less than the Quantity Ordered on one or more lines.  Due to the posting rules noted above, this distinction can only be made by evaluating the lines of the PO ... IMH and JCH can't be used.

The following registry entry changes the definition for open Credit PO’s to be those where the Quantity Received is less then Quantity Ordered:

Setting Key: CreditPOsMustBeReceived  Setting Value: true

The default value is false so as to preserve existing behavior. 

EEI Support Note: When the new setting is set to true, the behavior matches 'OkToBillWithOpenCreditPOs=false'. See case for details.

Technical Details:

When above information is entered in the wizard, the system will calculate regular, overtime and double time pay and bill hours based the following rules and write to the Labor tab of the work order:

Any hours worked before the Technician Start Time or after the Technician End Time defined in the Technician record will be considered overtime pay and default the calculated hours and Technician Overtime Pay Type Id found in the Technician record and other entered information to the Labor entry screen on the Work Order.

Any hours worked before the Regular Billing Start Time or after the Regular Billing End Time found in the Job referenced in the Work Order will be considered overtime billing and default the calculated hours and the Technician Overtime Labor Rate Id found in the Technician record along with other entered information to the Labor entry screen on the Work Order.

If any of the hours worked are on Saturday, they will be considered overtime pay and overtime bill unless the referenced Job has the "Saturday Included" box checked which will then calculate overtime pay but regular time bill if hours are within Reg. Billing Start/End times defined in Job record. If any of the hours worked are on Sunday, they will be considered overtime pay and overtime bill unless the referenced Job has the "Sunday Included" box checked which will then calculate overtime pay but regular time bill if hours are within Reg. Billing Start/End times defined in the Job record.

If any of the hours worked are on a defined company holiday or a Sunday, they will be considered double time pay and double time bill hours.

If Travel check box is marked, default the appropriate Travel Labor Rate Id from Technician Record.

If Shift Override check box is marked, treat hours worked as regular time.

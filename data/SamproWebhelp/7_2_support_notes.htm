3856  WO Import Costs Wizard

Code was implemented in case 2412 that examines qualifying WO Reguired Resource lines and already-recorded WO INV lines and imports discrepancies.  The decision is based on comparing Requisitioned Quantities to WO INV Quantities at the Item Id and Location Id level.  When the Requisitioned Quantity exceeds the WO INV Quantity, a new INV line is imported to account for the difference.

Encon has the following situation on a regular basis ... the Quantity Used (as recorded in Tech Anywhere) is less than the Quantity Requisitioned.  As noted above, in this scenario, the Import Wizard recognizes the difference and imports the additional quantity required to match up to the requisitions.

Per Tim's request, this behavior is being changed as follows ... instead of resolving quantity differences, the import wizard simply looks for WO-INV Item/Location combinations that match up with requisitions.  Quantity differences are no longer relevant.  This includes the situation where a WO-INV line exists for a given Item/Location combination and the line's Quantity is zero.

That last statement is important because current Tech Anywhere posting rules will post zero-quantity lines that are uploaded from a tech device.  This permits the tech to indicate that zero were used.

3427 Work Order viewer and Credit PO’s

One of the major differences between regular and credit POs is the posting of the Quantity Ordered and Quantity Received values.  On a credit PO, the Quantity Received is ignored and the Quantity Ordered is used in its place.  For example, on a regular PO the Quantity Received is posted to IMH as Quantity On-hand and to JCH as Actual Quantity.  For credit POs, the on-hand and actual quantities are derived from the Quantity Ordered.

Some customers may wish to use Quantity Received as an indicator that the vendor has actually credited them for the returned items rather than whether the Credit PO is vouchered.  By their definition, an open credit PO is one where the Quantity Received is less than the Quantity Ordered on one or more lines.  Due to the posting rules noted above, this distinction can only be made by evaluating the lines of the PO ... IMH and JCH can't be used.

The following registry entry changes the definition for open Credit PO’s to be those where the Quantity Received is less then Quantity Ordered:

Setting Key: CreditPOsMustBeReceived  Setting Value: true

The default value is false so as to preserve existing behavior.

The Accounting Period table is queried for eligible Accounting Periods.  To qualify, the period must be Active and its date range must include the defaulted Effective Date.  When multiple periods are eligible, the one with the smallest Period Id is chosen to become the new Accounting Period for the current transaction.  If no eligible period is found, nothing is changed.

3074 Archive Workorders

The two new functions, Start Archive Work Orders and Start Autoarchive Work Orders are designed to allow high-volume customers to set older Work Orders to an Archived status. The Update Function may need to be run to access these functions. To be eligible for archiving, the Work Order’s Date Closed must be set, and the Escalation Status would typically be set to Billed or Cancelled.

When Work Order is Archived, a Tech line will be added set to the Log Type of Archive with will override all other statuses when setting the Escalation Status and Status Category. To ‘unarchive’ a Work Order, the user can simply delete or change the ‘archived’ Tech line and save the Work Order.

Archived Work Orders will not appear on the Dispatch Board, but are still listed in the New Call WO pane.

A new button, Archived, has been added to the Work Order List; clicking this button filters the list to Archived Work Orders only.

1. Start Autoarchive Work Orders

This function is designed to be run as a scheduled event. The following registry options must be in effect.

Registry setting parameters

Id: sys-start-Sautoarchivewo

Setting Key: DaysOld Setting Value: #days

If this was set to 180 days, the work order would have to have a date closed on it 180 days or earlier than today.

EscalationStatuses=billed,cancelled

This is the recommended setting. Archiving ‘completed’ work orders with this function is NOT recommended as they may have unbilled costs.

 2. Start Archive Work Orders

 This is the manual process to archive work olders.

Required Fields: Date Opened / Date Closed: at least one of these fields would have to be completed.

 Ranges:

  a. Site ID

  b. Job ID

  c. Work Order ID

  d. Site Status (e.g, if site is inactive, archive the wo's)

  e. Escalation Status

Warning: use with caution the first time – this just hammers the CPU the first time you run it.

A Dictionary of useful (hopefully) information from the user's current session is compiled.  The top level of this Dictionary contains the following key-value pairs ...

The result is that FieldTableCursor may return multiple rows where only a single row is actually present in the table. Since the Report Writer relies on FieldTableCursor, it can error out when accessing the notes.text column.

In this release, this has been resolved by using DBResultSet to access the database as it does not exhibit this problem.

2644 GL Report Writer

In previous releases, in a GL Report Row Layout definition, a DefineTot row defines a total account whose sign depends on whether the computed total is a net Debit or a net Credit. Net Debits generate a positive total and net Credits generate a negative total. This is regardless of the value of the Sign Control field on the DefineTot row. This results, for example, in a net Income total always generating a negative total. This can lead to percentages of net income showing up with an improper sign.

This release implements the Sign Control field on the DefineTot row so that the report writer can define whether the computed total should be considered a normally Debit or a normally Credit account.
